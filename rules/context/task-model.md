# 컨텍스트 관리 지침 — 태스크 모델 (task-model · 파일시스템)

태스크를 **파일시스템 백엔드**에 어떻게 표현·저장하는지 정의합니다. 백엔드와 무관한 운영 규율·상태 전이 절차는 짝이 되는 `task-ops` 지침을 따릅니다.

이 프로젝트는 **파일시스템 백엔드**를 사용합니다. 모든 태스크는 `.tasks/` 아래에 마크다운 파일로 저장하고 git에 커밋합니다.

## 태스크 파일

- 위치: `.tasks/T-NNN.md`
- ID: `T-NNN` 일련번호. 한 번 부여하면 바꾸지 않습니다. 빈 번호는 재사용하지 않습니다.
- 한 파일 = 한 태스크. 디렉토리 중첩 없이 평평하게 둡니다.
- 부모-자식 관계는 frontmatter `parent:`로만 표현합니다.

### Frontmatter 스펙

```yaml
---
id: T-001
title: <짧은 제목>
status: in_progress       # 상태 집합: backlog, in_design, in_progress, blocked, review, done, cancelled
parent: T-000             # 상위 태스크 ID (없으면 생략)
depends_on: [T-002]       # 차단 의존 (없으면 생략)
owner: <agent-or-user>    # 진행 중·차단일 때만 채움 (선택)
created: 2026-04-28T10:00:00Z
refs:                     # 태스크 전체와 연관된 핵심 산출물 1~3개 (없으면 생략)
  - path/to/file.ts
  - https://github.com/.../pull/42
---
```

수정 시각·마지막 활동은 git 메타데이터(`git log`)로 확인합니다. frontmatter에 별도 필드를 두지 않습니다.

### 본문 섹션 (순서 고정)

```markdown
## 목표
측정·확인 가능한 결과 상태를 1~3문장으로 작성.

## 배경
문제·상황·리스크를 처음 보는 사람도 이해할 만큼 작성.

## 제안
접근, 구현 순서, 도구·라이브러리, 대안과 선택 이유를 이어서 실행 가능하게 작성.

## 검증 계획
테스트·도구·관찰 지표·수동 단계로 DoD 확인 방법을 작성.

## 완료 기준 (Definition of Done)
- [ ] ...
- [ ] ...

## 진행 로그 (append-only)
- 2026-04-28T10:00:00Z @agent-a: 시작. ...
- 2026-04-28T11:00:00Z @agent-a: ... 발견.

## 결정 (Decisions)
- 2026-04-28: X 대신 Y 선택. 이유: ...
```

- 목표는 무엇, 제안은 어떻게, DoD는 끝났는지의 체크리스트입니다.
- 검증 계획은 DoD를 확인하는 방법입니다.
- 진행 로그에는 결정·차단 사유·놀라운 발견·외부 영향이 있는 변경만 적습니다.
- `refs:`는 태스크 전체와 관련된 핵심 산출물 1~3개만 둡니다. 임시 참조는 진행 로그에 적습니다.

## 상태 집합과 백엔드 매핑

이 프로젝트가 사용하는 상태 집합: backlog, in_design, in_progress, blocked, review, done, cancelled

- 각 상태는 태스크 파일 frontmatter의 `status:` 값으로 표현합니다. 위 집합에 없는 값은 쓰지 않습니다.
- 상태는 `status:` 한 곳으로만 표현하고 별도 라벨·디렉토리로 중복 표시하지 않습니다.
- 상태가 진행하는 순서와 각 라이프사이클 이벤트의 목표 상태(전이 이벤트)는 `task-ops`가 정의합니다.

## 공통 규칙

- 태스크 파일은 git에 커밋합니다.
- 시각은 ISO 8601 UTC를 사용합니다 (예: `2026-04-28T10:00:00Z`).
- ID는 한 번 부여하면 바꾸지 않습니다. 잘못 만들었으면 취소 상태로 처리하고 새 ID로 다시 만듭니다.
