# 컨텍스트 관리 지침 — 운영 (task-ops)

AI 에이전트 간 작업 컨텍스트를 영속화하기 위한 **백엔드 무관 운영 규율**입니다. 세션 기억에 의존하지 않고 다음 에이전트가 이어받을 수 있어야 합니다. 태스크를 어디에 어떻게 저장하는지(파일·Issue 등)는 짝이 되는 `task-model` 지침이 정의합니다. 이 문서는 그 저장 방식과 무관하게 항상 성립하는 절차만 다룹니다.

## 태스크 우선 원칙 (no-task-no-work)

비자명한 작업은 **관련 태스크가 있을 때만 시작합니다**. 없으면 먼저 태스크를 만듭니다. 최초 등록(backlog 성격)은 제목 + 1-2줄 설명만으로 충분하며, "작다"·"즉흥적"이라는 이유로 생략하지 않습니다.

### 작업 시작 신호

- 첫 `Edit`/`Write` 호출
- `git add`/`git commit` 호출
- 외부 상태를 바꾸는 도구 호출 (외부 API, 인프라 변경, 태스크 백엔드 변경 등)
- 한 카테고리에서 다파일 변경을 시작할 때

신호 발생 시 태스크가 없으면 작업 중단 → 태스크 생성(저장 방식은 `task-model` 참조) → 작업 재개. 사용자에게 생성 사실을 알립니다.

### 예외

- 사용자가 명시적으로 "이번엔 태스크 없이"·"룰 무시"처럼 면제해준 경우.
- 단순 정보 조회(파일 read, `git log`, 목록 조회, 검색)는 작업 시작 신호가 아닙니다.

### 계획 섹션 채움

- 최초 등록은 제목 + 1-2줄 설명만 있어도 됩니다.
- 본격 설계·진행 상태로 전이할 때는 목표·배경·제안·검증 계획·완료 기준을 채웁니다. 빈 섹션이나 `TBD`로 전이하지 않습니다.
- 짧아도 명확해야 합니다. 시작 후 알게 된 사항은 진행 로그·결정에 추가하고, 필요하면 계획 섹션도 갱신합니다.

### 위반 발견 시

작업 중 태스크 누락을 발견하면 즉시 멈추고 retroactive 태스크를 만듭니다. 진행 로그의 `[handoff]` 항목에 사유를 남긴 뒤 재개합니다.

## 상태와 전이

기본 흐름: backlog → in_design → in_progress → review → done

이 흐름에 포함되지 않는 보조 상태(예: 차단·취소 성격)는 사유를 기록한 뒤 진입/복귀합니다. 사용 가능한 상태 집합과 각 상태의 백엔드 표현은 `task-model`이 정의합니다.

상태 전이는 **작업 에이전트의 책임**입니다. 아래 라이프사이클 이벤트가 발생하면 직접 상태를 갱신합니다. 이벤트 목록은 고정이며, 각 이벤트의 목표 상태는 이 프로젝트가 정한 값입니다.

| 라이프사이클 이벤트 (고정) | 목표 상태 |
|---|---|
| 계획/스펙 문서 생성 | in_design |
| 구현 시작 (첫 `Edit`/`Write`, `git add`/`git commit`) | in_progress |
| 검증 통과 (DoD 모두 체크 + 검증 PASS + 변경 반영) | review |
| 머지/완료 (PR 머지 또는 사용자의 명시적 "완료"·"Done"·"닫아줘" 신호) | done |
| 차단 발견 | blocked (진행 로그에 `[blocked]` 항목으로 사유 기록) |
| 차단 해제 | in_progress |

머지/완료 전이는 사용자 명시 신호 또는 머지가 있을 때만 수행합니다.

## 운영 규칙

1. **시작 전**: 관련 태스크를 확인하고 없으면 만듭니다. 대상과 마지막 `[handoff]`를 먼저 읽습니다.
2. **시작 시**: 상태를 진행 중으로, 담당(owner/assignee)을 본인으로 설정하고 시작 기록을 남깁니다.
3. **진행 중**: 의미 있는 발견·결정·차단을 즉시 기록합니다. 계획이 바뀌면 갱신합니다.
4. **commit 전**: 자동화 가능한 검증 계획을 모두 실행합니다. 실패하면 commit하지 말고 수정 후 재검증합니다. 수동 검증은 `[handoff]`에 담당자·시점을 남깁니다.
5. **완료 시**: DoD 점검 → 머지/완료 전이 → `[handoff]` 추가 → 담당 비움.
6. **append-only**: 진행 로그·결정·comments는 삭제·수정하지 않습니다. 정정은 새 항목으로 추가합니다.

## 태스크 단위

- 1 태스크 ≈ 한 에이전트가 한 세션에 끝낼 수 있는 분량.
- 더 크면 하위 태스크로 분할합니다. 부모 태스크는 하위 태스크의 진행 상황을 요약해 둡니다.
- 깊이는 2~3 단계 이내를 권장합니다.

## 기록 규율

- 진행 로그·결정에는 결정·차단 사유·놀라운 발견·외부 영향이 있는 변경만 적습니다. 모든 도구 호출이나 사소한 시도는 적지 않습니다.
- 결정은 `[decision]`, 인계는 `[handoff]`, 차단은 `[blocked]` prefix를 항목 첫 줄에 둡니다. 라벨이 아니라 본문 마커입니다.
- 시각·작성자는 백엔드가 자동 기록하면 그것을 사용하고, 아니면 ISO 8601 UTC로 적습니다 (예: `2026-04-28T10:00:00Z`).
